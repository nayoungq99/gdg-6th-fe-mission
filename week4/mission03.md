# 상태 관리의 필요성
## Props Drilling
리액트는 데이터가 부모 --> 자식인 단방향 흐름.  
최상위 컴포넌트에 있는 유저 정보가 최하위 컴포넌트에게 필요하다면  
중간에 있는 부모, 자식 컴포넌트들은 자신에게 필요 없어도 오직 전달만을 위해 **Props**를 받아서 넘겨줘야 함.   
--> Props Drilling

- **유지보수**: 중간 컴포넌트의 Props 이름을 하나 바꾸면 밑에 있는 컴포넌트가 전부 터짐.
- **불필요한 리렌더링**: 데이터가 바뀔 때 Props를 전달만 하던 중간 컴포넌트들까지 같이 리렌더링되어 성능이 떨어짐.
- **데이터 파편화**: 여러 컴포넌트에서 같은 데이터를 수정하다 보면 어느 순간 데이터의 일관성이 깨짐.

## Context API
리액트는 Props Drilling을 해결하기 위해 내장 기능인 Context API를 제공함.  
데이터가 흐르는 전역적인 방을 만들고 필요한 컴포넌트가 ``useContext``을 사용해  
중간 컴포넌트를 거치지 않고 데이터를 직접 꺼내 쓰는 방식.

## Context API가 있는데 왜 라이브러리 사용?
치명적인 단점 존재!
- **불필요한 전역 리렌더링(성능)**: Context 방에 있는 데이터(예: ``{ user, theme }``) 중 ``user``만 바뀌어도 ``theme``만 갖다 쓰던 컴포넌트까지 전부 강제로 리렌더링. (State 분리가 까다로움)
- **보일러플레이트(반복)**: Context를 쓰려면 ``Provider`` 만들고 ``value`` 설정한 뒤 상위 컴포넌트를 감싸는 과정이 꽤나 번거로움.

## 전역 상태 관리 라이브러리
- ``Zustand``: Flux (중앙 집중), 필요한 상태만 쏙 골라내어 리렌더링 최소화. 보일러플레이트 없음.
- ``Jotai / Recoil``: Atomic (원자 단위), useState처럼 아주 작은 단위(Atom)로 상태를 쪼개어 관리. 리렌더링 최적화 최고.
- ``Redux (Toolkit)``: Flux (중앙 집중), 엄격한 단방향 흐름, 강력한 디버깅 툴. 하지만 코드량이 압도적으로 많음.

### Zustand
- ``useState`` 만큼 직관적이라 10분만에 배울만큼 쉬움!
- 현재 다운로드 수가 상승 중이며 따라서 레퍼런스가 많음.
- 현재 하고 있는 쇼핑몰 프로젝트 정보를 담기에 가볍고 완벽함.
